1. Storing Videos in the Browser Cache (Blob Pre-fetching)
How it works:

Normally, browsers stream video in chunks. Every time you scroll up and down, it requests chunks again from Vercel.
The Solution: We can use JavaScript to fetch the entire video file as a Blob (Binary Large Object) on initial load, cache it in the browser's Cache Storage, and convert it into a local object URL (blob:http...).
When the <video> tag plays, it reads the video directly from the device's RAM/Disk cache with zero network requests and zero Vercel bandwidth usage.
2. Eliminating Initial Scroll Lag on Low-End Devices
How it works:

Standard videos are compressed using "keyframes" (I-frames) every few seconds. When you scroll, the phone's chip has to calculate the missing frames in between on-the-fly, which causes stuttering on the first scroll.
The Solution:
Once the Blob is cached, we can run a silent pre-decode routine in the background that quickly seeks through the video timeline once (video.currentTime = 1, then 2, then 3, etc.) to force the hardware decoder to parse and cache the decoded frames in the GPU's memory.
For the best result, the video file itself should be compressed with a high keyframe rate (a keyframe every 2–5 frames instead of every 120 frames), making seeking effortless for mobile chips.

How it works:

Changing the src attribute of a single <video> element forces the browser to destroy the old decoder, re-establish a connection, fetch the new video, and start decoding. This creates a black screen or layout lag.
